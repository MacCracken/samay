## 2. Threat model

**Attacker.** A party who can supply either (a) a **cron expression** string, or (b) a **serialized
snapshot** (JSON) that samay restores. The snapshot path is the primary concern: it reconstructs
scheduler state — task IDs, priorities, node capacities, epoch-ns timestamps, cron entries — and if
snapshots can arrive from disk or the network, every scalar and string in them is attacker-chosen.

**Trust boundaries.**
- *Cron expressions* cross a validation boundary at `cron_expr_parse`. This boundary is **strong**
  (see Findings / Verified-safe): every hostile expression tested is rejected at parse time.
- *Snapshots* cross the `*_from_json_str` boundary. This boundary is **weak**: unlike the Rust
  oracle's serde deserialize (which rejects missing/out-of-range fields), the port's deserializers
  accept partial and out-of-domain input and construct half-initialized structs. **Every confirmed
  finding lives here.**

**Cyrius-specific hazards weighed.** No integer-overflow traps, no bounds checks, no GC; heap-pointer
structs via `alloc`; `cstr`/`Str` with `0` as the null/None sentinel; manual `load8`/`store8`. The
absence of overflow traps and bounds checks makes the deserializer's missing-validation the highest-
impact class, because a stored `0`-sentinel `Str` faults on later dereference.

**In scope:** memory-safety (null-deref, OOB, overflow-into-alloc), algorithmic-complexity DoS,
unbounded work/memory driven by attacker counts, semantic corruption of the deterministic-scheduling
contract. **Out of scope:** privilege escalation, network/RPC surface, side channels, the Rust oracle.

## 3. Threat-landscape context (2024–2026)

We reviewed the current advisory record for software of samay's shape and mapped each class onto the
code. Four classes dominate; two were samay's historical near-misses and are now provably closed, two
remain live and account for the findings below.

### 3.1 Unbounded cron range/step expansion → memory/CPU exhaustion — **NOT susceptible (by design)**

- **CVE-2026-48593** (oban_web) — `"0 0 1-100000000 * *"` materialized a ~100M-element list
  (~2.4 GB), crashing the node.
  <https://cna.erlef.org/cves/CVE-2026-48593.html>
- **CVE-2024-43688** (OpenBSD cron) — `set_range()` heap underflow via a bad step value → memory
  scribble → root.
  <https://www.sentinelone.com/vulnerability-database/cve-2024-43688/>
- **croniter** unbounded-range DoS (healthchecks.io fuzz, `"0-1000000000 * * * *"`) — ~10 s CPU
  then crash.
  <https://blog.healthchecks.io/2023/03/i-found-a-vulnerability-in-croniter/>

The recurring root cause is that a *sibling* helper validates the range while the *expansion* helper
trusts the raw numbers. samay is structurally immune: `_cron_parse_term` (`src/cronexpr.cyr`) both
validates (`rlo>=lo`, `rhi<=hi`, `rlo<=rhi` at lines 127–129; `step>0` at line 107) **and** expands
in the same function, stores each field as a fixed **i64 bitmask** (not a materialized list), and
rejects numeric tokens >9 chars (`_cron_atoi`, line 55). The fill loop runs at most the field width
(≤60 iterations) and `1<<v` never shifts past `1<<59`. The OpenBSD step-underflow guard (`step<=0`
rejected) is **load-bearing in a language with no bounds checks — keep it.**

### 3.2 JSON number-parsing algorithmic-complexity DoS — **NOT applicable (remediated in pinned toolchain)**

- **CVE-2025-11230** (HAProxy / embedded mjson) — hang on crafted exponents.
  <https://windowsforum.com/threads/haproxy-cve-2025-11230-dos-mitigate-mjson-numeric-parsing.392491/>
- **CVE-2025-40929** (Cpanel::JSON::XS), **CVE-2025-57052** (cJSON, CVSS 9.8) — number-parsing
  memory-safety / DoS.

This is the exact class of the *known historical bayan bug* the brief asked to verify (the
`{"x":1e100000000}` O(exponent-value) apply loop). **Confirmed fixed in this tree:** vendored
`lib/bayan.cyr:2163–2172` documents the per-unit loop was replaced by a Grisu2/Clinger+DiyFp path
with the exponent **clamped/saturated during scanning** (`if (ev < 100000)`, `bayan.cyr:2718`),
applied via a cached-power table. Shipped as bayan 1.2.1 / cyrius 6.4.69; `cyrius.cyml` pins 6.4.69,
so the fixed parser is in tree. Probed through the real `NodeCapacity.total_cpu` f64 field:
`1e100000000`, `1e999999999`, `1e-999999999` all parse in <1 ms vs an 87 ms/100M busy-loop baseline.

### 3.3 Deeply-nested JSON uncontrolled recursion → stack overflow — **NOT susceptible (depth cap enforced)**

- **CVE-2025-52999** (jackson-core), **CVE-2024-58264** (serde-json-wasm), **CVE-2024-21907**
  (Newtonsoft.Json), **CVE-2025-53864** (Nimbus JOSE+JWT), **CVE-2026-48512** (MessagePack-CSharp),
  **CVE-2026-50193** (jackson-databind).
  <https://www.herodevs.com/blog-posts/cve-2025-52999-denial-of-service-via-stack-overflow-in-jackson-core>

samay's deserializers feed untrusted JSON straight into bayan's DOM parser. The parser enforces a
hard depth cap: `_JP_MAX_DEPTH = 128` (`lib/bayan.cyr:3304`), checked *before* each recursive descent
in both the DOM path (`_jp_parse_value`, `bayan.cyr:3540`) and the streaming path (`_js_parse_value`,
`bayan.cyr:4060`); over-deep input fails with `"nesting too deep"` (returns 0), which samay's
`*_from_jsonv` treat as `v==0` → clean reject. **Directly probed:** depths 128 accepted, 129 through
2,000,000 all rejected in constant time, exit 0, no segfault. This is exactly the recommended fix
(a hard nesting limit; Jackson defaults to 1000, bayan to 128).

### 3.4 Untrusted deserialization (CWE-502) + integer overflow (CWE-190) + hash-DoS — **LIVE; source of the findings below**

- **CWE-502 / CWE-190** — CISA Feb-2025 buffer-overflow secure-by-design alert; Next.js
  infinite-loop CVE-2025-67779.
  <https://www.cisa.gov/news-events/alerts/2025/02/12/cisa-and-fbi-release-secure-by-design-alert-eliminate-buffer-overflow-vulnerabilities>
- **Length-prefix / declared-count → unbounded alloc** — MessagePack-Java **CVE-2026-21452**,
  Perl Storable **CVE-2026-57433**.
  <https://osv.dev/vulnerability/CVE-2026-21452>
- **32-bit size/index overflow** — jq **CVE-2024-23337**, **CVE-2026-32316**.
  <https://www.cvedetails.com/cve/CVE-2024-23337/>
- **Hash-collision algorithmic-complexity DoS** — Netty QUIC **CVE-2025-29908**, Kwik
  **CVE-2025-23020**, Node.js/V8 HashDoS advisory (March 2026).
  <https://nodejs.org/en/blog/vulnerability/march-2026-hashdos>

Mapping: the *declared-count* and *32-bit-overflow* variants **do not apply** — samay's
`*_from_jsonv` use fixed-size struct allocs (`alloc(112)`, `alloc(72)`), iterate the *actual*
`json_v_arr_len` rather than any self-declared count, and are i64 throughout (no 32-bit truncation).
Both were probed and refuted (see §6). But the broader CWE-502 class — **missing field validation**,
**unclamped/overflowing scalars**, **hash-collision blowup on restore**, and **unbounded restored
collection sizes** — is exactly where samay diverges from the Rust serde oracle, and is the source of
every confirmed finding.

## 4. Findings (confirmed only)

Ten verdicts were CONFIRMED by running probes; four candidate issues were refuted (§6). All confirmed
findings are on the **snapshot-restore path** and share one root cause family: the port's
deserializers do not enforce the required-field / domain constraints that Rust serde enforces for
free. Severity-ranked below.

### 4.0 Remediation status (v0.7.0)

Recommendations 1 and 2 — the ship-blockers in §8 — **landed in v0.7.0** ([ADR-0005](../adr/0005-restore-input-validation.md)),
and every fix was re-verified by re-running the finding's PoC (crash → clean reject/exit 0) plus a
permanent regression guard in `tests/samay.tcyr`.

| Finding | Severity | Status in v0.7.0 |
|---|---|---|
| F1 missing required `Str` → SIGSEGV on re-serialize | Medium | **FIXED** — `*_from_jsonv` reject the record (`return 0`) on a missing/non-string required field |
| F2 missing map-key field → null-key `str_eq(0,0)` SIGSEGV at restore | Medium | **FIXED** — same rejection keeps null-keyed records out of the maps |
| F3 missing nested `resource_requirements` → null-deref on schedule | Medium | **FIXED** — task/template with `req==0` is rejected |
| F6 unclamped `priority` i64-overflow on restore | Low | **FIXED** — clamped to 1–10 (and `status` range-checked) on restore |
| F7 negative resource values on `NodeCapacity` restore | Low | **FIXED** — `_jv_uint` clamps the u64-domain fields to ≥0 |
| F4 hash-collision DoS (unseeded FNV-1a) | Medium | **MITIGATED** — `SAMAY_JSON_MAX_ITEMS` (100000) caps restored collection size; the true fix (upstream stdlib hash seeding) is tracked, Rec 5 |
| F5 O(n²) insertion sorts on unbounded collections | Medium | **MITIGATED** — same size cap bounds N; the stable O(n log n) sort + terminal-task pruning is tracked as a follow-up, Rec 3 |
| F8 no aggregate cron work budget across entries | Low | **PARTIALLY MITIGATED** — entry count now bounded by the restore cap; aggregate budget + `last_fired` clamp tracked, Rec 4 |
| F9 SKIP-policy full 100000-iteration due-count | Low | **PARTIALLY MITIGATED** — entry count bounded by the restore cap; early-exit count variant tracked, Rec 4 |

The remaining follow-ups (Rec 3–5) are **not ship-blockers**: F4/F5's unbounded/OOM vector is closed
by the cap, and the residual within-cap cost is bounded DoS, not a crash or corruption. They are
deliberately deferred from this security release rather than risk a merge-sort rewrite destabilising
the M5 determinism guarantee; each is tracked in the roadmap.

The findings below describe each issue **as discovered** (pre-fix); the "Fix" line is what shipped.

> No **Critical** or **High** finding. The memory-safety crashes are deterministic denial-of-service
> (SIGSEGV / null-deref), not memory *corruption* — no attacker-controlled write, no code-exec
> primitive was found. They are rated **Medium**.

### F1 — Missing required `Str` field in a snapshot → null-`Str` deref (SIGSEGV) on re-serialization — **Medium**

- **Site:** `scheduled_task_from_jsonv` (`src/json.cyr:169–188`) via `_jv_str` (`src/json.cyr:43–48`);
  crashes downstream in `scheduled_task_to_jsonv` (`src/json.cyr:190–206`) →
  `json_v_str_new(0)` → `str_len(0)`/`str_data(0)` (`lib/str.cyr:56,71`) → `load64(8)`.
  Same pattern for every required `Str`: `ScheduledTask.{task_id,name,description,agent_id}`,
  `NodeCapacity.node_id` (`:245`), `CronEntry.name` (`:278`), `CronTaskTemplate.{name,description,agent_id}`
  (`:140–142`), `TrainingJobTemplate.{model_id,dataset}` (`:123,125`).
- **Vector:** a snapshot that simply omits a required string field. `_jv_str` returns `0` (null `Str`)
  for an absent/non-string key; the deserializer stores it verbatim with no validation. The Rust
  oracle's serde would return a "missing field" `Err` and reject the whole document.
- **PoC (running probe, real tree):**
  - Restore `{"tasks":[{"name":"b","priority":5}],"nodes":[]}` (task_id omitted) → **succeeds**;
    `task_scheduler_to_json_str(s)` → **exit 139 (SIGSEGV)**.
  - Control: the same doc with all required strings present → serializes cleanly, exit 0 — isolating
    the missing required `Str` (not the nested `resource_requirements`) as the cause.
  - Root cause confirmed in isolation: `str_len(0)` alone → exit 139 (reads `load64(8)`).
- **Fix:** in each `*_from_jsonv`, never store `0` into a required `Str`. Preferred (matches Rust and
  the project's "validated at parse time" principle): add a `_jv_str_req` that signals absence so the
  deserializer **rejects the document** (`return 0`) when a required string is absent or non-string.
  Alternative: substitute `str_from("")`. Add a roundtrip test restoring a snapshot with omitted
  required fields and re-serializing without crashing.

### F2 — Missing map-key field → null `Str` key → `str_eq(0,0)` SIGSEGV **during restore** — **Medium**

- **Site:** `task_scheduler_from_jsonv` `map_set(..., ScheduledTask_task_id(t), t)` (`src/json.cyr:418`);
  same at `:428` (`NodeCapacity.node_id`) and `:369` (`CronEntry.name`). Fault in
  `_map_key_eq` → `str_eq(0,0)` → `load64(a+8)` (`lib/str.cyr:164`).
- **Vector:** two array elements that both omit the field used as the map key. `_jv_str` yields key
  `0`; `hash_str_v(0)` is guarded to `0`, so both null keys hash to bucket 0; the second insert calls
  `str_eq(0,0)`, dereferencing address 8. Crashes *at parse/restore time*, before any save.
- **PoC (running probe):** `task_scheduler_from_json_str("{\"tasks\":[{\"name\":\"a\",\"priority\":5},{\"name\":\"b\",\"priority\":5}],\"nodes\":[]}")`
  → prints "before parse", then **exit 139** during the parse. Control with both tasks carrying a
  `task_id` returns cleanly.
- **Fix:** reject entries whose required key field is missing before inserting — e.g.
  `var tid = ScheduledTask_task_id(t); if (t != 0 && tid != 0) { map_set(...); }` at `:418`, and the
  same non-zero-key guard at `:428` and `:369`. Keeping null-keyed objects out of the maps also
  prevents F1's serialize-side crash. (Cannot fix in `lib/str.cyr` — vendored; the deserializer is
  the correct layer.)

### F3 — Restored task with missing nested `resource_requirements` → null-deref on schedule — **Medium**

- **Site:** `scheduled_task_from_jsonv` stores `_rr_from(...)` verbatim (`src/json.cyr:178`);
  `_rr_from` returns `0` for the absent key (`src/json.cyr:83`). Same store in
  `cron_task_template_from_jsonv` (`src/json.cyr:156`). Fault reached via
  `task_scheduler_schedule_pending` → `_best_fit_node(s, req=0)` → `node_capacity_can_fit(node, 0)`
  → `ResourceReq_cpu_cores(0)` (load at address 0).
- **Vector:** a snapshot whose task omits `resource_requirements`. Restore succeeds with a `0`
  pointer; the first schedule pass after load faults.
- **PoC (running probe):** restore a 1-task/1-node snapshot with `resource_requirements` omitted →
  `s!=0`, 1 task, `resource_requirements==0`; `task_scheduler_schedule_pending(s)` → **exit 139**.
  Control with the field present → 1 scheduling decision, exit 0.
- **Fix:** when `_rr_from` returns `0`, substitute `resource_req_default()`, or reject the snapshot
  (matching serde required-field semantics). Apply at both `src/json.cyr:178` and `:156`.

### F4 — Hash-collision algorithmic-complexity DoS on snapshot restore (unseeded FNV-1a) — **Medium**

- **Site:** `lib/hashmap.cyr` `hash_str_v` is an **unseeded** FNV-1a (offset basis
  `0xCBF29CE484222325`, prime `0x100000001B3`, no per-process seed) with open-addressing linear
  probing. Reached by `task_scheduler_from_jsonv` (`src/json.cyr:408–420`, keyed by attacker task_id),
  and equally `cron_scheduler_from_jsonv` (`:359–373`, keyed by `CronEntry_name`) and the nodes map
  (`:422–431`, keyed by `node_id`).
- **Vector:** because the hash has no per-process seed, a colliding key set is **precomputable once
  and works against every samay instance**. Bulk-inserting colliding keys degrades `map_set` to
  O(n²).
- **PoC (running probes, timer validated against a 50M busy-loop ≈ 450 ms):**
  - Raw map, N=8000: colliding keys 493 ms vs distinct 2.5 ms (~197×).
  - Real entry point `task_scheduler_from_json_str`, identical parse cost, only task_id contents
    differ: N=8000 (182 KB doc) → 1.00 s colliding vs 43 ms distinct (~23×); N=16000 (370 KB) →
    2.44 s vs 79 ms (~31×); denser collisions at final capacity → 1.42 s vs 37 ms (~38×).
  - A ~185 KB snapshot ≈ 1.4 s single-core CPU; a multi-MB crafted snapshot scales to tens of
    seconds / minutes.
- **Fix (layered):** (1) samay-side — **bound restore volume**: cap `json_v_arr_len` for
  `tasks`/`nodes`/cron entries in `task_scheduler_from_jsonv` / `cron_scheduler_from_jsonv` (reject
  snapshots exceeding a documented limit) and/or reject oversized documents before parse; this caps N
  so O(n²) stays bounded and is shippable without touching `lib/`. (2) Upstream stdlib — seed
  `hash_str_v`/`hash_str` with a random per-process value, or move to a keyed/collision-resistant
  hash (SipHash-style). Seeding is the real fix; the count bound is what samay can ship now.
- **Note:** samay's "deterministic scheduling / explicit sort" principle does **not** mitigate this —
  that rule governs output *iteration order*; the blowup is at *insert* time inside `map_set`.

### F5 — O(n²) insertion sorts over unbounded, attacker-influenced collections — **Medium**

- **Site:** four insertion sorts — `_sort_tasks` (`src/scheduler.cyr:42`), `_sort_tasks_by_id`
  (`:63`), json `_sort_by_key` (`src/json.cyr:318`), cron `_sort_entries_by_name` — over collections
  with no size cap anywhere. `submit_task` (`scheduler.cyr:89`) only `map_set`s; `cancel_task`
  (`:113`) only flips status and **never removes from the map**, so the working set grows
  monotonically and every `pending_tasks`/`schedule_pending`/`to_json_str` re-sorts the whole map.
- **Vector:** `task_scheduler_from_json_str` rebuilds the tasks/nodes maps from an unbounded untrusted
  JSON array; each subsequent serialize re-sorts via the quadratic `_sort_by_key`.
- **PoC (running probe, timer validated against a 91 ms busy-loop):** N equal-priority queued tasks
  (tie-break → random UUID = average case), timing `task_scheduler_pending_tasks`:
  N=1000→5 ms, 2000→23 ms, 4000→93 ms, 8000→369 ms, 16000→1471 ms — clean ~4×/doubling = O(n²).
- **Fix:** replace the four insertion sorts with an O(n log n) **stable** sort (merge sort preserves
  the ordering the comments rely on). Independently, prune terminal-state tasks
  (CANCELLED/COMPLETED/FAILED) from the map instead of only flipping status, and bound collection
  sizes in `task_scheduler_from_jsonv` (shared with F4's cap).

### F6 — Reconstructed `priority` is unclamped and silently i64-overflows on restore — **Low**

- **Site:** `scheduled_task_from_jsonv` stores `_jv_int(v,"priority",1)` directly (`src/json.cyr:177`)
  with no 1–10 clamp — unlike `scheduled_task_new` (`src/types.cyr:191–200`). Same unclamped pattern
  in `cron_task_template_from_jsonv` priority (`src/json.cyr:155`).
- **Vector:** bayan `_jp_atoi` does `n = n*10 + digit` with no overflow trap; a 30-digit all-INT token
  wraps silently.
- **PoC (running probe):** `{"priority":999999999999999999999999999999}` → `ScheduledTask_priority ==
  5076944270305263615` (silent i64 wrap); `{"priority":-5}` → `-5` accepted verbatim.
  Constructor `scheduled_task_new(...,999,...)` correctly clamps to 10.
- **Impact:** **not memory-unsafe** — priority feeds only comparisons and a branchy classifier
  (`task_priority_from_numeric`), no alloc/index/loop bound. The damage is a broken
  deterministic-scheduling contract: a `~5.07e18` task sorts ahead of every legitimate task;
  negatives sort last.
- **Fix:** clamp on restore as the constructor does — `if (p<1) p=1; if (p>10) p=10;` before
  `set_priority`, at both `src/json.cyr:177` and `:155`. (Rust uses `u8`; serde already rejects
  out-of-range/negative.)

### F7 — Negative resource values accepted on `NodeCapacity` restore (u64 domain divergence) — **Low**

- **Site:** `node_capacity_from_jsonv` (`src/json.cyr:248–252`) reads `total_memory_mb`,
  `available_memory_mb`, `total_disk_mb`, `available_disk_mb`, `running_tasks` via `_jv_int` with no
  range check. Rust oracle (`rust-old/src/lib.rs:202–205`) types these `u64`/`usize`; serde rejects a
  negative and fails the whole restore.
- **Vector:** a snapshot with negative memory/disk. Values stored verbatim (probe read back
  `total_memory_mb=-5000`, `available_memory_mb=-9000`, `disk=-1`).
- **Impact:** input-validation divergence only — **no crash/overflow**. The candidate "satisfies any
  request" narrative was **refuted** by the same probe: `node_capacity_can_fit(node, 256MB)` returned
  `0` (a negative-available node refuses *all* tasks — the opposite of over-admission), and
  `node_capacity_utilization` returned `0.0` with the divisions guarded by `if (total>0)` (no
  divide-by-bogus-total). The "over-admit" behavior only reproduces with large *positive* values,
  which are valid `u64` and accepted by Rust too — not a divergence.
- **Fix:** validate the unsigned-domain fields on restore — add a `_jv_uint` helper that rejects
  (or clamps to 0) a negative `i64` for these five fields, restoring parity with the Rust oracle.

### F8 — Cron catch-up scan has per-entry caps but **no aggregate work budget** across entries — **Low**

- **Site:** `cron_scheduler_check_due_at` entry loop (`src/cron.cyr:209`). Per-entry defenses hold
  (window clamp, `CRON_MAX_COUNT`, `CRON_CATCHUP_CAP`), but there is no global cap across entries, and
  the entry count is itself unbounded via snapshot restore. `last_fired` is read straight from JSON
  (`src/json.cyr:296`) with no clamp.
- **Vector:** a snapshot with many enabled, rarely-matching cron entries carrying a stale
  `last_fired`. Each forces a full look-back scan on the first post-restore check.
- **PoC (running probe, timer validated against a 100 ms busy-loop):** entries with expr `"0 0 1 1 *"`
  and 100-year-stale `last_fired`: 1 entry=136 ms, 10=1270 ms, 40=5111 ms, 100=12672 ms — perfectly
  linear at ~127 ms/entry, no aggregate cap. 100 crafted entries (~25 KB JSON) ⇒ ~12.7 s single-
  threaded blocking on the first check. Per-entry caps worked as designed (only 2 tasks fired/entry).
- **Fix:** add a shared work budget across the entry loop (`CRON_SCAN_TOTAL_BUDGET` of total
  `_cron_count_due` iterations per check; on exhaustion, clamp remaining entries' `last_fired` to
  `now - CRON_SCAN_WINDOW_SECS` and log). Alternatively, clamp `last_fired` older than the catch-up
  window up to `now-window` **at restore** — the pre-window occurrences are dropped-and-logged anyway,
  so firing semantics are unchanged.

### F9 — Cron SKIP-policy computes the full 100000-iteration due-count for a log line — **Low**

- **Site:** under `CRON_SKIP`, `_cron_count_due` runs to `CRON_MAX_COUNT=100000` matches
  (`src/cron.cyr:162`) even though the count is used only for the `_cron_log_skip dropped N` line
  (`src/cron.cyr:231–233`) while a single task fires.
- **Vector:** many restored entries with a matching-heavy expr (e.g. `"* * * * *"`) and stale
  `last_fired`; linear-amplified by the uncapped restored entry count.
- **PoC (running probe, timer validated against an 88 ms busy-loop):** single `_cron_count_due("* * *
  * *", ...)` returned due=100000 in ~29.4 ms; a 200-entry scheduler under SKIP took 5.859 s
  (~29.3 ms/entry), firing exactly one task/entry. ⇒ ~5 min at 10k entries. Work is bounded and
  terminates; no crash. (Correction to an early hypothesis: the burst is one-time per restore, not
  per tick — `check_due_at` sets `last_fired=now` after each check.)
- **Fix:** under SKIP the exact count is never needed (the branch fires once, logs `due-1`). Add an
  early-exit count variant that stops as soon as a second match is found (distinguish 0/1/≥2),
  collapsing per-entry SKIP cost from ~29 ms to microseconds. Combine with F4/F8's restore-count
  bound and `last_fired` clamp.

## 5. Verified safe

Each item below was checked and found **not** vulnerable — most by running a probe against the real
tree.

- **bayan number-parser exponent DoS** (the brief's headline "known issue"): **NOT present**.
  `lib/bayan.cyr:2163–2172` replaced the O(exponent-value) apply loop with a Clinger/DiyFp path;
  the exponent saturates at `ev<100000` (`bayan.cyr:2718`). `1e100000000`, `1e999999999`,
  `1e-999999999` all parse in <1 ms vs an 87 ms/100M baseline. (bayan 1.2.1 / cyrius 6.4.69, pinned.)
- **Deeply-nested JSON recursion**: hard cap `_JP_MAX_DEPTH=128` enforced before descent in both
  parsers (`bayan.cyr:3540,4060`); depth 129…2,000,000 rejected cleanly, no stack overflow.
- **CWE-502 count-overflow / undersized-alloc / OOB**: no deserializer reads a self-declared
  count/size; all array loops bound by the *actual* `json_v_arr_len`; all struct allocs are fixed
  constants (`alloc(112)`=14×8, `alloc(72)`=9×8, `alloc(48)`=6×8) — no `count*size` multiply to
  overflow. Probed max-i64 scalars and a 2000-node array → no crash, no OOB.
- **i64 epoch/duration overflow in restored `last_fired`**: fully mitigated by the
  `CRON_SCAN_WINDOW_SECS` clamp, the `t <= end` loop bound, `CRON_MAX_COUNT`, and `CRON_CATCHUP_CAP`.
  Probed i64-max/min and wrap-literal timestamps → zero-iteration or clamped/capped paths, no wrap,
  no negative-epoch, no crash.
- **Unbounded cron range/step expansion**: `_cron_parse_term` validates (`rlo>=lo`, `rhi<=hi`,
  `rlo<=rhi`, `step>0`) and expands in the same function; field is an i64 bitmask; `1<<v` bounded to
  `1<<59`. A 400 KB pathological field parsed in 8 ms (linear, not quadratic). `*/0`, `0/0`, `60`,
  `999999999999`, `-5`, `,,,,,`, `*5`, `@bogus`, trailing comma — all rejected at parse, no crash.
- **`_cron_atoi` overflow guard**: rejects fields >9 chars (`cronexpr.cyr:55`), capping accumulation
  at 999,999,999 — no i64 overflow in field accumulation, no garbage read.
- **cron negative-epoch guard**: `cron_expr_matches` returns 0 for `epoch_secs<0`
  (`cronexpr.cyr:259`), avoiding `1<<-1`.
- **Struct alloc sizes vs field counts**: all constructors and the two hand-alloc `from_jsonv` paths
  match layout exactly (ResourceReq 48/6, ScheduledTask 112/14, NodeCapacity 72/9, plus CronExpr,
  CronEntry, SchedulerStats, etc.) — no setter writes past its allocation, no heap overflow.
- **uuid buffer bounds**: `b[16]` indexed 0..15; `out=alloc(37)` = 32 hex + 4 dashes + NUL;
  `_UUID_HEX` indexed by `(byte>>4)`/`(byte&0x0F)`, both 0..15. No OOB.
- **cstr vs Str discipline** in the JSON layer is consistent (`json_v_obj_get` cstr keys,
  `json_v_obj_set` Str keys, `json_v_parse_str(str_data,str_len)`) — no Str-as-cstr core-dump in
  samay's own code.
- **Type confusion** (string/object/array where int/obj expected): coercers return benign defaults
  (`json_v_int/float` → 0/0.0, `json_v_str` → 0 on non-STR, `json_v_arr_*` bounds-checked). Probed
  `resource_requirements` as a string and `tasks` as an object → no crash.
- **ai-hwaccel `profile_from_json`**: guards `is_obj`, uses `PROFILE_NONE/0` defaults for every field
  — no crash on malformed/partial profile objects.
- **Duplicate JSON keys**: `json_v_obj_get` returns the first match deterministically — no last-wins
  UB.
- **Scheduler stats i64 overflow** (extreme restored timestamps): overflow is real but effect is
  **wrong-output only** (garbage `average_*_ms`); wrapped values flow only into accumulators and
  divisions by a constant / guarded denominators — no load8/store8, no index, no div-by-zero. Not a
  memory-safety vuln (see §6).
- **`cron_expr_next_after`** (the one ~5.26M-iteration scan with a potential `t+3.16e8` overflow) has
  **no caller** in `src/` — not reachable from any untrusted path.

## 6. Inconclusive / follow-ups

- **Do snapshots actually cross a network/disk trust boundary in deployment?** Every confirmed
  finding is scoped "attacker-controlled" *conditional on* snapshots being restorable from an
  untrusted source. If, in samay's deployment, snapshots are only ever produced and consumed
  in-process by trusted code, F1–F9 drop from Medium/Low to hardening items. **A human should confirm
  the snapshot provenance / trust boundary** — it is the single fact that sets the severity of this
  audit. The report assumes the conservative case (untrusted snapshots).
- **Upstream hash seeding (F4).** The real fix (seed `hash_str_v` or move to SipHash) lives in the
  vendored `lib/`, which samay must not modify. This needs an upstream Cyrius stdlib issue/decision;
  samay can only ship the restore-count bound as interim mitigation.
- **Refuted candidates (recorded so they are not re-litigated):** i64 stats overflow (wrong-output,
  not memory-unsafe); deep-nesting recursion (cap enforced); CWE-502 count/alloc overflow (no
  declared-count pattern exists); i64 epoch overflow in restored timestamps (clamped/capped). Each was
  driven to a running probe and observed safe.

## 7. Recommendations (prioritized)

1. **Enforce required-field and domain validation at the deserializer boundary (fixes F1, F2, F3,
   F6, F7).** This is the central fix and restores parity with the Rust serde oracle. Add
   required-field helpers (`_jv_str_req`, `_rr_from` non-null substitution) that **reject the whole
   document** (`return 0`) when a required string / nested struct / map-key is absent or the wrong
   type, and clamp scalars to their declared domain (`priority` 1–10; memory/disk/running_tasks ≥ 0).
   Aligns with the project's stated "validated at parse time" principle. Add roundtrip tests that
   restore snapshots with omitted/negative/overflowing fields and must reject-or-survive without a
   crash.
2. **Bound restored collection sizes (mitigates F4, F5, F8, F9 in one move).** Cap `json_v_arr_len`
   for `tasks`/`nodes`/cron-entries in `task_scheduler_from_jsonv` / `cron_scheduler_from_jsonv`, and
   reject oversized snapshot documents before parse. This is the defense samay can ship without
   touching `lib/`, and it caps every quadratic/linear-amplification path at a known N.
3. **Replace the four O(n²) insertion sorts with a stable O(n log n) merge sort (F5)**, and prune
   terminal-state tasks from the map instead of only flipping status, so the working set cannot grow
   unbounded across a long-lived scheduler.
4. **Clamp `last_fired` to `≥ now - CRON_SCAN_WINDOW_SECS` on restore, and add an aggregate cron work
   budget + a SKIP early-exit count (F8, F9).** Pre-window occurrences are dropped-and-logged anyway,
   so firing semantics are unchanged.
5. **File an upstream Cyrius stdlib issue to seed `hash_str_v`/`hash_str` (F4).** The unseeded FNV-1a
   collision set is precomputable once against all instances; seeding is the only true fix.
6. **Defense-in-depth (no confirmed finding, still worth it):** keep the `step>0` cron guard
   (load-bearing in a bounds-check-free language — its removal reintroduces CVE-2024-43688);
   keep the bayan depth cap and exponent clamp pinned by holding `cyrius=6.4.69` (do not float the
   toolchain without re-verifying these two mitigations); and consider saturating subtraction in the
   stats math so extreme restored timestamps yield 0 rather than wrapped averages.

---

## 8. Conclusion

samay's **parser boundaries are strong** and its **historically-dangerous classes are provably
closed**: the cron parser is structurally immune to the 2024–2026 unbounded-expansion CVE cluster
(oban_web, croniter, OpenBSD cron), and the bayan JSON parser in the pinned toolchain has both the
exponent-DoS clamp and the nesting-depth cap that the current jackson/mjson/cJSON advisories are
about. No memory *corruption*, no OOB write, and no code-execution primitive was found; the Str
migration, uuid buffer, struct allocations, and cron arithmetic are all memory-safe under adversarial
probing.

The residual risk is **entirely in the snapshot-restore path**, where the port diverges from the Rust
serde oracle by accepting partial and out-of-domain input. This yields three deterministic
null-deref DoS crashes (F1–F3), two algorithmic-complexity DoS vectors (F4 hash-collision, F5
quadratic sort), and semantic/amplification issues (F6–F9). All are **Medium/Low** — denial-of-service
and correctness, not corruption — and all are conditional on snapshots arriving from an untrusted
source.

**Ship verdict:** the ship-blocking recommendations — Rec 1 (required-field / domain validation) and
Rec 2 (bound restored collection sizes) — **landed in v0.7.0** ([ADR-0005](../adr/0005-restore-input-validation.md);
see §4.0). Every confirmed crash (F1–F3) is closed and re-verified by re-running its PoC to a clean
reject/exit-0, plus permanent regression guards; the input-validation divergences (F6, F7) are closed;
and the unbounded/OOM vector of the DoS findings (F4, F5, F8, F9) is capped by `SAMAY_JSON_MAX_ITEMS`.
The residual within-cap cost of F4/F5/F8/F9 is bounded DoS (not crash or corruption) and is tracked as
Rec 3–5 for a follow-up release — deliberately deferred rather than risk a sort rewrite destabilising
the M5 determinism guarantee. **v0.7.0 is fit to ship** from a security standpoint; the snapshot
trust-boundary question in §6 remains the one operational input a deployer should confirm (it only
lowers, never raises, the residual severity).
